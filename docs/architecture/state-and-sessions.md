# State and sessions

## Authority and renderer projection

The SDK-host `AgentSession` is the sole live-state authority. `state-authority.mjs` takes direct reads from public SDK getters and publishes snapshots with host identity, epoch, sequence, queues, catalog, and revisioned editor state. It does not derive state from `agent_*`, retry, or transcript events.

`SessionRegistry` accepts only snapshots for the active host identity and increasing epoch/sequence. It publishes `available`, `transitioning`, or `unavailable` runtime state with a transport-only lease whose timing does not vary with Pi liveness. Availability is an independent transport fact: the renderer applies it even when main repeats the current snapshot for diagnostics. On lease expiry or a transport gap it fences stale state and resyncs; UI selectors must treat unavailable state as unknown, not idle or a completed turn. Sidebar completion status requires a fresh available snapshot that ends streaming on the same host and epoch.

Lifecycle admission is child-owned. Before reload, worktree respawn, or narrow activation-visit release, main requests the host's serialized opaque lifecycle permit. The child samples direct SDK getters and custody, intents, queues, editor, and extension UI in one authority transaction and returns only an allowed/reason verdict. Main rechecks only process identity, transport availability, and local lifecycle fences after awaiting it; reload repeats the child check at its transition boundary. Retained intents are retired by explicit child dispositions/outcomes, never because main observes an idle snapshot.

The Zustand session store is a projection of this authority plus renderer-only presentation state (transcript blocks, drafts, overlays, notifications, and panel buffers). Transcript events render history and activity; they do not define runtime liveness.

## Authority-frame target and compatibility migration

[ADR 0002](../decisions/0002-authority-frames-and-plane-synchronization.md) defines the planned replacement for the compatibility projection described in the remaining sections. It deliberately does **not** promise that renderer state equals Pi state at the same instant. A `following` plane is exact only as of its named authority cursor—`(hostInstanceId, sessionEpoch, transportSequence, snapshotSequence)`—and contains all complete frames received through that cursor. A child mutation after the cursor may be unknown to the renderer until publication arrives.

The target reducer owns one `PlaneSync` state per semantic control, transcript, extension-UI, and panel plane: `following(cursor)`, `synchronizing(lastCursor, reason)`, or `unavailable(lastCursor, reason)`. A sequence gap fences only the affected plane; it does not turn unrelated following planes into guesses. While synchronizing, retained values are diagnostics visibly marked stale and controls requiring that plane are fenced. No plane returns to following from silence or unrelated events.

Each child authority commit emits an atomic semantic frame: ordered records plus a complete terminal semantic snapshot at one owner/cursor. The renderer reduces the whole frame in one store commit and never combines records or snapshots from different cursors. High-volume transcript deltas and terminal panel bytes have their own plane cursors and cannot mutate semantic control state. Owner replacement atomically installs a successor baseline and rejects every predecessor publication.

Attach and resync are baseline-plus-replay operations. Main buffers publications for the renderer generation, requests a child-serialized baseline after prior authority work, installs it at a high-water mark, then replays only the contiguous buffered tail. The baseline includes the semantic snapshot, operation-journal watermark, extension-UI reconstruction state, transcript live-tail/overlap boundary, and panel keyframes or explicit repaint requirements. Buffer overflow or a gap requests another baseline rather than pretending a current snapshot repairs continuity. ANSI chunks alone cannot reconstruct a panel: it is synchronizing and non-interactive until a framebuffer keyframe or acknowledged full repaint; state that Pi cannot reconstruct after child death is closed as unavailable.

Every target mutation has a stable ID, expected owner, and child-written journal entry before possible Pi dispatch. Duplicate same-owner IDs with the same payload execute once; a conflicting reuse is rejected. Receipts report admitted/duplicate/not-admitted/delivery-unknown, while typed intent outcomes arrive in frames and alone update canonical operation state. A lost post-dispatch settlement is permanently `outcome_unknown` review custody and is not replayed against a successor. Observed operation records cover agent, retry, bash, navigation, command, and compaction state machines; a compact command invocation is `command`/`invoking` plus a conservative barrier, never a fabricated compaction start. Bounded journal and dispatched-intent retention expose independent low/high watermarks and truncation without retaining terminal payload/image bytes or claiming coverage before a truncation watermark.

Target replacement freezes child semantic ingress, obtains a main advisory-lock permit for the target, then publishes one terminal successor frame before predecessor routing is released. Lock contention rejects mutable activation; the advisory lock coordinates cooperative Pi-Vis owners and is not a security boundary. This keeps old-owner frames and delayed outcomes from changing the successor projection.

This is a staged migration, not a claim that all current data structures already have these semantics. The existing identity/epoch, snapshot sequence, lease/resync, dispositions, revisioned editor, generation/UI acknowledgement, panel sequence, and transition-batch protocols remain supported compatibility behavior. During rollout they must be bridged at explicit baselines; mixed records must never be presented as one cursor. The renderer semantic slice is no longer shadow mode: `RendererAuthorityState.authoritativeSnapshot` is the sole semantic projection while its semantic plane is `following`. Direct runtime snapshots, transition batches, and disposition receipts are compatibility diagnostics or presentation inputs only; on synchronization loss, retained authority data is `staleDiagnosticSnapshot`, controls are fenced, and it cannot fall back to a legacy snapshot. The remaining sequence is child serialization/journal, main buffered per-plane publication, mutation slices, presentation reconstruction, then removal of legacy mutation/lifecycle paths.

## Submissions and queues

All text/image work is submitted through `session.submit`. A submission identifies the expected host and epoch, the editor revision it was composed against, intent id, queue mode, UI surface, text, and images. Every valid submission attempt returns a correlated disposition, including main-process fences before host dispatch (closing, unavailable, missing, or stale identity). Malformed IPC input alone is rejected as an error. The host returns a disposition:

- `not_submitted` — identity or editor revision was stale;
- `in_custody` — a compaction/navigation barrier retained the request;
- `admitting` — a diagnostic preflight/admission deadline elapsed but this child still owns the prompt promise; it remains pending and later emits exactly one terminal disposition;
- `consumed` — pi accepted prompt admission;
- `rejected` — admission failed before consumption;
- `extension_error` — an invoked extension command failed and is treated as consumed rather than replayable;
- `completed` — a previously accepted request settled;
- `outcome_unknown` — host could not establish the result, including a prompt failure after successful preflight crossed the possible-consumption boundary.

The renderer changes drafts, attachments, and pending UI only in response to these outcomes. `admitting` is pending evidence, not a terminal unknown outcome. Composer suppresses only an exact duplicate in-flight effective payload; same text with changed files/images remains a distinct intent with its own disposition. The host clears authoritative editor text and attachments at the first revision-matched custody/consumption acknowledgement; a timed-out but still child-owned admission remains `admitting`, and its later terminal acknowledgement performs the same guarded clear while preserving a newer editor revision. If a host fails after dispatch, custody, consumption, or completion of an active-turn queue admission, main publishes the exact retained text and images as an idempotent review restoration before replacement; ambiguous work is never represented by a toast alone. Completion carries an explicit queued-admission marker, so an idle turn that safely completed is not spuriously restored. Custody is FIFO and drains before new ingress only after an explicitly successful barrier; cancellation, abort, retry failure, or navigation error converts affected custody into review-required restoration instead of submitting it. If an earlier idle prompt promise remains unresolved after streaming ends, later ingress receives immediate `in_custody` rather than holding its main-process RPC open, then drains only when that prompt fence settles. Nested navigation barriers restore their own failed/cancelled records; whenever the outermost navigation settles, any remaining eligible inner-barrier custody is drained rather than stranded. Queue text arrays in each snapshot are direct SDK values. Parallel positional intent arrays identify GUI-owned queue slots without treating extension-transformed text as identity, and the host decorates the corresponding delivered user event with that intent so the renderer transfers visible ownership exactly once. Queue admission claims only one slot relative to a freshly synchronized baseline; ambiguous external mutations invalidate correlation rather than stealing ownership. ESC queue restoration carries cleared GUI intents so their optimistic transcript ownership transfers atomically to the review card. ESC during active streaming asks the host to clear/abort before falling back to an unresolved-preflight result; the resulting restoration preserves queued texts and original attachment ownership. Active-turn image submissions that time out during admission enter the attachment ledger conservatively until direct queue state proves they can be retired. The renderer shows restoration text and originals in a separately dismissible review card with an explicit warning that extensions may have transformed them. Restoration IDs are idempotent, never overwrite newer draft text, and are retired from main only after the user acknowledges review. Acknowledging an `ambiguous-submission:<intentId>` restoration atomically retires its retained intent as well, stopping replay and releasing planned-respawn eligibility.

## Renderer generations, UI, and panels

Renderer attachment has a monotonic generation. Main asks the host to cancel old-generation blocking UI and only proceeds after its acknowledgement. Dialog answers and panel close requests carry operation ids and are acknowledged by the host. Panel input uses a per-panel sequence acknowledgement; main serializes each panel's in-flight inputs before validating gaps, and detected gaps are repair signals. Authoritative panel close/clear/reset events retire the renderer sequence key, so a replacement host may safely reuse its process-local panel IDs while ordinary same-host React remounts preserve sequencing. Composer/editor synchronization is revisioned: text plus unsent image/file attachment payloads are patched into host-owned editor state and survive renderer reload/close review. Non-prompt command dispatch first waits for the revisioned patch that produced its editor text. Recognized commands carry a request id and captured host/epoch through IPC, are rejected rather than queued across replacement, and clear both the mounted Composer and authoritative editor after a correlated `completed` settlement—even when the operation reports a surfaced domain error such as “nothing to compact”—but only while the originating component instance and local edit generation still match and the captured runtime remains current, ready, and available. Effectful and replacement commands additionally retain an intent before dispatch. If child IPC rejects the command it settles `not_executed` and preserves input; if IPC accepted it but the host acknowledgement is lost it settles `outcome_unknown`, is never replayed, and publishes an explicitly acknowledged interrupted-command review. Built-in picker continuations retain the runtime identity that opened the picker, including model changes, fork/scope/logout/trust commands, follow-up model-catalog reconciliation, and trust-triggered reload. `/reload` is identity-bound independently and main revalidates its captured host after the fresh admission snapshot before issuing the in-place reload. Input that never dispatches, conflicts during editor synchronization, or belongs to a stale unmounted completion is preserved. After an editor-patch transport failure, unchanged retry first fetches and validates a fresh authoritative runtime snapshot, rebases to its editor revision, and only then retries the patch. Accepted prompts tolerate the host's custody-clear revision advance but preserve later renderer edits and later host injections. Pending-session promotion migrates any newer workspace draft into the real session before retiring the workspace slot. Extension `setEditorText` and `pasteToEditor` mutate that same host revision; main treats their UI message only as a resync notification, never an unversioned injection. These extension mutations preserve unresolved conflict candidates and their attachment sets until explicit reconciliation. If a local patch is already in flight, the renderer suppresses replacement until the host reports acceptance or exposes both values in explicit Composer conflict review. Image reads acquire renderer custody synchronously before `FileReader` starts, so an attachment-only pending session cannot be reaped during the read and remains recoverable in the sidebar after switching away. Each read captures the session/editor generation; completion after a session or host change is discarded with a warning rather than attaching private data to the new context. Unified clipboard reads similarly capture panel/editor revision and text: an unchanged reply advances replicated editor state, while an intervening edit or retired TUI generation preserves the temp image as an attachment/conflict instead of silently merging it into a newer draft or dropping it. Submit is refused while a read is outstanding, and completion patches the current text ref rather than the text captured when the picker opened. A patch is accepted only from its base revision, otherwise the host returns the canonical text/attachments and conflict data. Rejection advances a local patch epoch: every already-queued optimistic patch is fenced, the revision rebases to the host, and only a subsequent user edit may explicitly reconcile the preserved versions. Unified-editor submit requests carry the host revision captured at Enter, and unresolved requests remain replicated in main and host across renderer reattachment until an authoritative response retires them. Main gives a claimed request a bounded claim ID/deadline; expiry creates one non-replayable restoration and stable-intent tombstone, releases the host-side pending editor snapshot without injecting ambiguous text, and fences every late renderer continuation or response. A retired `(host, epoch, request id)` remains tombstoned for the session record, so a duplicate late host record receives a refusal instead of creating a new claim or intent.

## Lifecycle

1. **Open:** `session.open` creates a registry record; history stays on disk until requested.
2. **Activate:** `session.activate` forks the SDK host and acquires the advisory lock. For a pathless new session, the child discovers the SessionManager file and waits for an `initial_session_file_permit`; main must reserve that exact file before initial extension binding or ready/following publication. Contention is a hard startup failure. Main pauses its pre-ready watchdog for correlated dialogs/custom panels while retaining a longer bounded human-response deadline. It then receives initial binding records plus their terminal direct snapshot in one ready commit before becoming available. A startup failure is failed/unavailable; there is no alternate runtime.
3. **Run:** snapshots and host records keep queue, liveness, catalog, and editor state current. Every awaited command disposition/completion publishes a fresh direct snapshot; asynchronous prompt preflight and terminal settlement each publish their own boundary. Extension UI and panel records preserve host ordering. Lifecycle watchdog time is suspended only by dialog/custom-panel promises opened in that lifecycle operation's async context; unrelated persistent UI cannot wedge a transition. Extension command-context `newSession`, `fork`, and `switchSession` actions use the same replacement transition/fatal-boundary path as renderer commands. Every replacement takes a fresh direct snapshot and refuses to begin while unrelated SDK work, submission admission/prompt fences, custody, or another host command remains active. A supported extension command-context replacement carries its owning submission intent through async context and may exclude exactly that one admission—never unrelated intents, commands, custody, navigation, or compaction—from the guard; `waitForIdle` polls the fresh public session getter under a bounded deadline. Host command ingress is synchronously counted and rejected once a transition starts, closing the before-dispatch/transition race. Reload is admitted only after main obtains an available correlated idle snapshot and the host rechecks fresh idle/submission/custody getters immediately before starting its transition. Full-state probes wait for transition settlement rather than exposing provisional snapshots, and invalidation/reload cancel dialogs that belong to the retiring extension generation. Provisional blocking dialogs and custom panels create main-process control-watchdog and registry snapshot leases that end only on host acknowledgement, panel close, or atomic transition commit—not merely when main sends a response—so normal human think time and delayed host processing are not misclassified as host silence. The correlated lifecycle command/reload deadline is suspended by the same lease and resumes with only its unspent active-execution budget.
4. **Transition/reload:** runtime replacement is an epoch transition. Before replacement work begins, the host emits a sequenced transition-start boundary; main immediately publishes `transitioning`, and host admission independently returns `not_submitted` while the transition exists. Cancellation stages its old-epoch full snapshot while availability remains `transitioning`, then emits a correlated settlement boundary that atomically restores available old-epoch state. Terminal transition batches must match both the pending transition ID and its announced provisional epoch, and the terminal snapshot must be fresher than prior same-host state; malformed, unsolicited, stale, or regressed batches cannot clear the transition fence. Submission terminal records retain their immutable admission host/epoch. A forced predecessor settlement is emitted on the live transition channel rather than folded into the successor batch; main converts it to non-replayable review and never publishes it as successor completion. Editor custody and asynchronous editor-patch acknowledgement both revalidate record, process, host, epoch, availability, and transition state after awaiting, so even a forced late predecessor settlement cannot be attributed to or clear successor state. The host buffers transition-sensitive records and emits them with one terminal snapshot; the renderer reduces the entire batch and terminal snapshot in one atomic store commit. Reload is refused while the direct snapshot is non-idle.
5. **Failure:** main fences the old host state; resolves and cancels old-host UI acknowledgements; dismisses dialogs; clears panel maps, checkpoints, input chains, and renderer panels; marks uncertain retained work `outcome_unknown`; retains the last meaningful authoritative editor checkpoint; and can attempt one bounded SDK-host recovery. Before a planned respawn, main first requests a fresh correlated snapshot and repeats every idle/custody/retained-work eligibility check immediately before detach; before publishing any replacement host's initial batch, it rehydrates retained text/attachments and recreates conflict review state. Rehydration validates the editor disposition and retries against a concurrently advanced replacement revision rather than treating transport success as acceptance. If bounded retries all conflict, the live replacement remains authoritative and the old checkpoint—including attachments—stays as its explicit conflict candidate; successful recovery likewise retains replacement-side text and attachment objects for review. Runtime editor conflict state can carry alternate and additional text-and-attachments candidates so pre-existing checkpoint conflicts are not collapsed when recovery introduces another candidate. Candidate identity is the `(text, attachments)` pair, so equal text with distinct files/images remains separately reviewable, while candidates identical in both fields are shown only once across authoritative, local, retained, and replacement state. The renderer independently preserves any newer local draft or staged attachments that had not yet reached main. If the replacement also carries an older host conflict, review keeps three distinct candidates—the actual renderer draft, authoritative replacement text, and retained alternate—rather than allowing “Keep my edit” to select the wrong value. No old-identity UI operation can time out against the replacement host. Renderer editor patches, dialog responses, and panel input/resize/close carry the host instance and session epoch captured by their originating surface; main rejects them if either identity no longer matches, including when numeric panel IDs or editor revisions are reused. Panel render effects and input/acknowledgement sequence maps are keyed/reset by the full `(hostInstanceId, sessionEpoch, panelId)` identity, so even an old passive effect firing during a coalesced same-ID replacement cannot consume the new panel's first sequence. Planned worktree respawn applies the same UI retirement before detaching the old process and refuses to move while authoritative or retained work is active.
6. **Close:** `prepareClose` first freezes renderer ingress in main and then asks the authoritative host to freeze ingress and produce the review token/checkpoint. The checkpoint includes direct state, custody/intents/restorations, revisioned editor and unified submissions, dialogs, and bounded current panel state. `confirmClose` succeeds only when both host and main mutation sequences still match; failed confirmation cancels/reopens both boundaries. Already-admitted command, ESC, reload preflight, worktree restart, editor patch, and UI/panel acknowledgement paths revalidate record/process/runtime and the close freeze after every await; late work is fenced or unknown and cannot dispatch a replacement, mutate the checkpoint, or publish predecessor completion. Failed or cold records without a live host use a main-local token so their tabs remain closeable; `cancelClose` reopens both boundaries. Closing releases the lock; session files and worktrees remain on disk.

There is no fixed active-host cap and no generalized idle/LRU eviction. A renderer activation visit carries a unique token only when it causes a cold→live transition. Switching away within the two-second startup-cancellation window may release that exact token: main handles a release that races Pi discovery, waits for startup if necessary, takes a fresh direct snapshot, and retires the host through graceful disposal only if the visit had no user interaction and all submission, queue, editor, command, review, transition, dialog, panel, extension-published status/widget/working, and acknowledgement state is empty. The release-before-activation tombstone uses the same two-second expiry, so a delayed or failed discovery cannot retain cancellation state indefinitely. A pre-existing or older host is never assigned to the visit, returning cancels an in-flight release, and an unsafe release is abandoned rather than retried as a broad idle policy.

## Workspace search isolation

Workspace search is persisted-history-only and is not a second runtime authority. A disposable worker index may catalog JSONL for a currently registered workspace and known worktrees, including all saved branches and pre-compaction history, but JSONL supplies results and exact context. Querying and read-only preview do not touch `SessionRegistry`, activate a host, or change active workspace/session, branch, transcript, draft, attachments, editor revision, unread state, or runtime identity. Context is occurrence-exact and reports `ready`, `relocated`, `changed`, `removed`, `forbidden`, or `unavailable`; it never substitutes changed content.

Only explicit result opening enters normal stored-session opening and its activation-visit lifecycle. There remains no fixed active-host cap or generalized idle/LRU policy; search cannot reap a pre-existing host. See [workspace session search](session-search.md).

## History and transcript

Session discovery and complete history hydration read persisted JSONL asynchronously. Initial hydration waits for an idle runtime and retries if the live transcript changes during the read, preventing persisted/live overlap from duplicating or overwriting racing events. Persisted transcript hydration walks the complete active on-disk branch across compaction markers: compaction limits the model's active context but does not remove earlier GUI scrollback. Main caches only a small LRU working set of converted complete histories so multiple large sessions do not remain resident indefinitely. Every `session.loadHistory` request carries the expected file, renderer history generation, and explicit host/epoch fields: a live runtime supplies both values, while an explicitly cold owner supplies a required `null`/`null` pair. Main admits the cold-owner form only for a non-activating record with no process or runtime identity. For live reads it requires ready/available, non-transitioning host/epoch identity, captures the actual registry record, process object, file, host, and epoch before file I/O, and validates the same lifecycle owner afterward; cold reads are therefore invalidated if activation begins before or during I/O, and live reads are invalidated as soon as transition starts. The renderer independently increments history ownership on file adoption, history seeding, and host/epoch replacement, and applies the complete history only while generation, file, and runtime identity still match. Initial open retries only after a store subscription observes a concrete live owner (or a genuinely new cold generation); it never retries an unchanged cold-owner request on an arbitrary short timer. Once ownership is stable, hydration waits for idle and rereads whenever the transcript changes during I/O. Each ownership/idle wait has a bounded timeout, and terminal failed/exited state ends the wait. The transcript reducer turns pi events and persisted entries into blocks, including custom entries and cache notices. Successful compaction moves the live block array by reference into immutable archive chunks and starts a small mutable tail at the marker; complete hydrated history is also retained as an archive chunk. This preserves complete, immediately available GUI scrollback while keeping normal per-token copies and reconciliation confined to post-hydration or post-compaction live-tail work. It is intentionally separate from runtime authority: history loading, tree navigation, and transcript rendering cannot overwrite a current host snapshot.
