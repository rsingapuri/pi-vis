# State and sessions

## Authority and renderer projection

The SDK-host `AgentSession` is the sole live-state authority. `state-authority.mjs` takes direct reads from public SDK getters, resolves getter/event disagreement inside the child, and serializes each semantic commit as an authority frame: ordered records plus a complete terminal snapshot with owner, epoch, and sequences. It does not derive semantic state from `agent_*`, retry, or transcript events.

`SessionRegistry` accepts only active-owner publications and routes frames without reducing or splitting them. Its identity/epoch checks, availability lease, and direct runtime snapshots are transport/compatibility diagnostics, not an alternate semantic projection. The host's periodic publication emits a semantic frame plus a direct control snapshot from the same child read: the frame advances canonical state, while only the control snapshot renews the independent availability lease. On a lease failure or discontinuity it fences the relevant authority plane and requests a baseline; UI selectors treat unavailable or synchronizing state as unknown, never idle or a completed turn. Sidebar completion requires a following semantic projection at its named owner/cursor. Owner-bound read-only `session.query` returns typed `ok`, `superseded`, `transitioning`, or `unavailable` data rather than throwing lifecycle races across IPC. `session.authorityAttach` likewise returns only typed `ready`, `transitioning`, or `unavailable` data; callers install an authority baseline only for `ready`.


Lifecycle admission is child-owned. Before reload, worktree respawn, or narrow activation-visit release, main requests the host's serialized opaque lifecycle permit. The child samples direct SDK getters and custody, intents, queues, editor, and extension UI in one authority transaction and returns only an allowed/reason verdict. Main rechecks only process identity, transport availability, and local lifecycle fences after awaiting it; reload repeats the child check at its transition boundary. Retained intents are retired by explicit child dispositions/outcomes, never because main observes an idle snapshot.

The Zustand session store is a projection of this authority plus renderer-only presentation state (transcript blocks, drafts, overlays, notifications, and panel buffers). Transcript events render history and activity; they do not define runtime liveness.

## Deployed authority-frame projection

[ADR 0003](../decisions/0003-authority-frames-and-plane-synchronization.md) defines the deployed projection. It deliberately does **not** promise that renderer state equals Pi state at the same instant. A `following` plane is exact only as of its named authority cursor—`(hostInstanceId, sessionEpoch, transportSequence, snapshotSequence)`—and contains all complete frames received through that cursor. A child mutation after the cursor may be unknown to the renderer until publication arrives.

The authority reducer owns one `PlaneSync` state per semantic control, transcript, extension-UI, and panel plane: `following(cursor)`, `synchronizing(lastCursor, reason)`, or `unavailable(lastCursor, reason)`. A gap in a known source-plane cursor fences only that plane; it does not turn unrelated following planes into guesses. A gap in main's single global publication sequence has unknown source-plane attribution, so it fences every plane until a serialized baseline is installed. While synchronizing, retained values are diagnostics visibly marked stale and controls requiring that plane are fenced. A fenced extension-UI plane retains its last status/widget presentation until its own following baseline replaces it; semantic publications cannot clear it. By [ADR 0005](../decisions/0005-retain-presentation-labels-across-authority-fences.md), a fenced semantic plane likewise retains last-known session/model/thinking labels as stale presentation while clearing dispatch/control fields. No plane returns to following from silence or unrelated events.

Each child authority commit emits an atomic semantic frame: ordered records plus a complete terminal semantic snapshot at one owner/cursor. The renderer reduces the whole frame in one store commit and never combines records or snapshots from different cursors. High-volume transcript deltas and terminal panel bytes have their own plane cursors and cannot mutate semantic control state. Owner replacement atomically installs a successor baseline and rejects every predecessor publication. Renderer and main may send only high-level, owner-bound intents; they may not choose Pi abort targets, prompt-versus-queue admission, or another Pi semantic result.

Attach and resync are baseline-plus-replay operations. Main buffers publications for the renderer generation, requests a synchronous child read-barrier baseline, installs it at a high-water mark, then replays only the contiguous buffered tail. The read barrier bypasses long-running mutation ingress; mutations that land afterward advance source cursors into replay. Same-owner/same-generation requests are single-flight, superseded requests cannot mutate router state after their awaits, and a successor renderer generation inherits any bounded same-owner attach tail so its baseline cannot cover and erase an unseen user event. The renderer atomically materializes transcript events in the replay as well as reducing their cursors, so a first user echo emitted during attach cannot disappear while a later assistant event remains. The baseline includes the semantic snapshot, operation-journal watermark, extension-UI reconstruction state, transcript live-tail/overlap boundary, and panel keyframes or explicit repaint requirements. Buffer overflow or a gap requests another baseline rather than pretending a current snapshot repairs continuity. ANSI chunks alone cannot reconstruct a panel: it is synchronizing and non-interactive until a framebuffer keyframe or acknowledged full repaint; state that Pi cannot reconstruct after child death is closed as unavailable.

Every mutation has a stable ID, expected owner, and child-written journal entry before possible Pi dispatch. Duplicate same-owner IDs with the same payload execute once; a conflicting reuse is rejected. Receipts report admitted/duplicate/not-admitted/delivery-unknown, while typed intent outcomes arrive in frames and alone update canonical operation state. A lost post-dispatch settlement is permanently `outcome_unknown` recovery custody and is not replayed against a successor. Observed operation records cover agent, retry, bash, navigation, command, and compaction state machines; a compact command invocation is `command`/`invoking` plus a conservative barrier, never a fabricated compaction start. Bounded journal and dispatched-intent retention expose independent low/high watermarks and truncation without retaining terminal payload/image bytes or claiming coverage before a truncation watermark.

Replacement freezes child semantic ingress, obtains a main advisory-lock permit for the target, then publishes one terminal successor frame before predecessor routing is released. Lock contention rejects mutable activation; the advisory lock coordinates cooperative Pi-Vis owners and is not a security boundary. This keeps old-owner frames and delayed outcomes from changing the successor projection.

The identity/epoch, snapshot sequence, lease/resync, dispositions, revisioned editor, generation/UI acknowledgement, panel sequence, and transition-batch protocols remain supported only as compatibility diagnostics or explicit-baseline bridges. They are non-authoritative: mixed records must never be presented as one cursor, repair a gap, settle a mutation, or overwrite the authority reducer. `RendererAuthorityState.authoritativeSnapshot` is the sole semantic projection while its semantic plane is `following`; on synchronization loss, retained authority data is `staleDiagnosticSnapshot`, controls are fenced, and it cannot fall back to a legacy snapshot. Semantic snapshots canonically carry `sessionName` (including a clear); rename outcomes do not derive it. Legacy mutation and presentation consumers are prohibited: no generic command channel, direct runtime snapshot, transcript event, receipt, returned promise, disposition, or lifecycle settlement may mutate canonical semantic state outside the child frame/reducer path.

## Submissions, custody, and silent draft recovery

All text/image work is submitted through `session.submit`. A submission names the expected host/epoch, editor revision, stable intent ID, queue mode, surface, text, and images. Every valid attempt receives a correlated disposition, including main-process lifecycle fences; malformed IPC is rejected as an error. `not_submitted`, `in_custody`, `admitting`, `consumed`, `rejected`, `extension_error`, `completed`, and `outcome_unknown` distinguish admission evidence from terminal authority-frame outcomes. `admitting` remains child-owned pending evidence, not a terminal unknown result. A mounted Composer may clear its matching visible draft as soon as correlated `in_custody`/`consumed` feedback proves host custody, but only under its originating owner/revision/local-edit fences; this presentation action never changes semantic authority or treats a generic intent receipt as completion. Uncertain/rejected work remains recoverable through the ordinary restoration path.

Custody is FIFO. A compaction or navigation barrier retains ingress and drains eligible custody before later ingress only after the barrier succeeds. Cancellation, failed navigation, interrupted queue delivery, and uncertain admission produce a child `queue_restoration` record with original text/attachments, cleared intent IDs, and certainty (`not_processed` or `unknown`). Host queue snapshots and host-correlated delivered user events preserve ownership without using transformed text as identity.

The record is not a renderer action request. Main resolves it to the one-way `session.restoreDraft` event: `not_processed` restores directly; `unknown` is reconciled against persisted session history at the retained dispatch offset; processed/command-only evidence resolves to `dropped`. The renderer applies each restoration ID once, merges a restoreable payload into the current draft and attachments without overwriting newer work, then calls `session.acknowledgeRestoration`. A dropped restoration is acknowledged after recording its informational outcome. There is no restoration channel or card, restore/dismiss action, or user-selected acknowledgement lifecycle.

This reconciliation is deliberately false-negative biased: only an exact persisted user-message match proves processing, so transformed, missing, malformed, unreadable, or otherwise inconclusive evidence can restore a redundant draft. It never causes automatic replay. A post-dispatch `outcome_unknown` is permanently non-replayable across replacement; restoration only gives the user a draft to inspect and submit, and never invokes Pi itself. See [ADR 0004](../decisions/0004-silent-reconciliation-replaces-user-review.md).

## Renderer generations, UI, and panels

Renderer attachment has a monotonic generation. Main asks the host to cancel old-generation blocking UI and only proceeds after its acknowledgement. An attach that crosses a cold or unresponsive host, closed IPC channel, session disposal, process replacement, same-process epoch replacement, or newer renderer generation resolves as typed unavailability, with full record/process/host/epoch/generation identity revalidated after cancellation and baseline awaits; already-issued Electron IPC cannot be revoked by renderer-side retry cancellation. Superseding a cancellation waiter retires it without letting its old timeout publish unavailability over the successful successor. A transport discontinuity rejects pending authority attachment immediately and retries only after its correlated state repair. Dialog answers and panel close requests carry operation ids and are acknowledged by the host. Panel input uses a per-panel sequence acknowledgement; main serializes each panel's in-flight inputs before validating gaps, and detected gaps are repair signals. Authoritative panel close/clear/reset events retire the renderer sequence key, so a replacement host may safely reuse its process-local panel IDs while ordinary same-host React remounts preserve sequencing. Composer/editor synchronization is revisioned: text plus unsent image/file attachment payloads are patched into host-owned editor state and survive renderer reload/close recovery. Non-prompt command dispatch first waits for the revisioned patch that produced its editor text. Recognized commands carry a request id and captured host/epoch through IPC and are rejected rather than queued across replacement. Intent-shaped commands clear both the mounted Composer and authoritative editor on child admission; that clear is final — a later failed terminal outcome is a domain error surfaced through the transcript and toasts and never reinjects the command text. Submit/invoke actions retain terminal clearing behavior: they clear only after their correlated accepted terminal outcome. All clearing is fenced to the originating component instance, local edit generation, and current ready/available captured runtime. Effectful and replacement commands additionally retain an intent before dispatch. If child IPC rejects the command it settles `not_executed` and preserves input; if IPC accepted it but the host acknowledgement is lost it settles `outcome_unknown`, is never replayed, and publishes an acknowledged dropped-restoration outcome. Built-in picker continuations retain the runtime identity that opened the picker, including model changes, fork/scope/logout/trust commands, follow-up model-catalog reconciliation, and trust-triggered reload. `/reload` is identity-bound independently and main revalidates its captured host after the fresh admission snapshot before issuing the in-place reload. Input that never dispatches, conflicts during editor synchronization, or belongs to a stale unmounted completion is preserved. An empty initial editor injection cannot overwrite a non-empty pending-workspace or per-session renderer draft: Composer rebases that exact draft once to the attached authoritative editor (including its current replicated attachments) through `session.editorPatch` only, never submission. After an editor-patch transport failure, unchanged retry first fetches and validates a fresh authoritative runtime snapshot, rebases to its editor revision, and only then retries the patch. Accepted prompts tolerate the host's custody-clear revision advance but preserve later renderer edits and later host injections. Pending-session promotion migrates any newer workspace draft into the real session before retiring the workspace slot. Extension `setEditorText` and `pasteToEditor` mutate that same host revision; main treats their UI message only as a resync notification, never an unversioned injection. These extension mutations preserve unresolved conflict candidates and their attachment sets until explicit reconciliation. If a local patch is already in flight, the renderer suppresses replacement until the host reports acceptance or exposes both values in explicit Composer conflict presentation. Image reads acquire renderer custody synchronously before `FileReader` starts, so an attachment-only pending session cannot be reaped during the read and remains recoverable in the sidebar after switching away. Each read captures the session/editor generation; completion after a session or host change is discarded with a warning rather than attaching private data to the new context. Unified clipboard reads similarly capture panel/editor revision and text: an unchanged reply advances replicated editor state, while an intervening edit or retired TUI generation preserves the temp image as an attachment/conflict instead of silently merging it into a newer draft or dropping it. Submit is refused while a read is outstanding, and completion patches the current text ref rather than the text captured when the picker opened. A patch is accepted only from its base revision, otherwise the host returns the canonical text/attachments and conflict data. Rejection advances a local patch epoch: every already-queued optimistic patch is fenced, the revision rebases to the host, and only a subsequent user edit may explicitly reconcile the preserved versions. Unified-editor submit requests carry the host revision captured at Enter, and unresolved requests remain replicated in main and host across renderer reattachment until an authoritative response retires them. Main gives a claimed request a bounded claim ID/deadline; expiry creates one non-replayable restoration and stable-intent tombstone, releases the host-side pending editor snapshot without injecting ambiguous text, and fences every late renderer continuation or response. A retired `(host, epoch, request id)` remains tombstoned for the session record, so a duplicate late host record receives a refusal instead of creating a new claim or intent.

## Lifecycle

1. **Open:** `session.open` creates a registry record; history stays on disk until requested.
2. **Activate:** `session.activate` forks the SDK host and acquires the advisory lock. For a pathless new session, the child discovers the SessionManager file and waits for an `initial_session_file_permit`; main must reserve that exact file before initial extension binding or ready/following publication. Contention is a hard startup failure. Main pauses its pre-ready watchdog for correlated dialogs/custom panels while retaining a longer bounded human-response deadline. It then receives initial binding records plus their terminal direct snapshot in one ready commit before becoming available. A startup failure is failed/unavailable; there is no alternate runtime.
3. **Run:** snapshots and host records keep queue, liveness, catalog, and editor state current. Every awaited command disposition/completion publishes a fresh direct snapshot; asynchronous prompt preflight and terminal settlement each publish their own boundary. Extension UI and panel records preserve host ordering. Lifecycle watchdog time is suspended only by dialog/custom-panel promises opened in that lifecycle operation's async context; unrelated persistent UI cannot wedge a transition. Extension command-context `newSession`, `fork`, and `switchSession` actions use the same replacement transition/fatal-boundary path as renderer commands. Every replacement takes a fresh direct snapshot and refuses to begin while unrelated SDK work, submission admission/prompt fences, custody, or another host command remains active. A supported extension command-context replacement carries its owning submission intent through async context and may exclude exactly that one admission—never unrelated intents, commands, custody, navigation, or compaction—from the guard; `waitForIdle` polls the fresh public session getter under a bounded deadline. Host command ingress is synchronously counted and rejected once a transition starts, closing the before-dispatch/transition race. Reload is admitted only after main obtains an available correlated idle snapshot and the host rechecks fresh idle/submission/custody getters immediately before starting its transition. Full-state probes wait for transition settlement rather than exposing provisional snapshots, and invalidation/reload cancel dialogs that belong to the retiring extension generation. Provisional blocking dialogs and custom panels create main-process control-watchdog and registry snapshot leases that end only on host acknowledgement, panel close, or atomic transition commit—not merely when main sends a response—so normal human think time and delayed host processing are not misclassified as host silence. The correlated lifecycle command/reload deadline is suspended by the same lease and resumes with only its unspent active-execution budget.
4. **Transition/reload:** runtime replacement is an epoch transition. Before replacement work begins, the host emits a sequenced transition-start boundary; main immediately publishes `transitioning`, and host admission independently returns `not_submitted` while the transition exists. Cancellation stages its old-epoch full snapshot while availability remains `transitioning`, then emits a correlated settlement boundary that atomically restores available old-epoch state. Terminal transition batches must match both the pending transition ID and its announced provisional epoch, and the terminal snapshot must be fresher than prior same-host state; malformed, unsolicited, stale, or regressed batches cannot clear the transition fence. Submission terminal records retain their immutable admission host/epoch. A forced predecessor settlement is emitted on the live transition channel rather than folded into the successor batch; main converts it to non-replayable `outcome_unknown` recovery evidence and never publishes it as successor completion. Editor custody and asynchronous editor-patch acknowledgement both revalidate record, process, host, epoch, availability, and transition state after awaiting, so even a forced late predecessor settlement cannot be attributed to or clear successor state. The host buffers transition-sensitive records and emits them with one terminal snapshot; the renderer reduces the entire batch and terminal snapshot in one atomic store commit. Reload is refused while the direct snapshot is non-idle.
5. **Failure:** main fences the old host state; resolves and cancels old-host UI acknowledgements; dismisses dialogs; clears panel maps, checkpoints, input chains, and renderer panels; marks uncertain retained work `outcome_unknown`; retains the last meaningful authoritative editor checkpoint while recovery is attempted; and can attempt one bounded SDK-host recovery. Before a planned respawn, main first requests a fresh correlated snapshot and repeats every idle/custody/retained-work eligibility check immediately before detach; before publishing any replacement host's initial batch, it rehydrates retained text/attachments and recreates conflict state. Rehydration validates the editor disposition and retries against a concurrently advanced replacement revision rather than treating transport success as acceptance. If bounded retries all conflict, the live replacement remains authoritative and the old checkpoint—including attachments—stays as its explicit conflict candidate; successful recovery likewise retains replacement-side text and attachment objects as conflict candidates. Runtime editor conflict state can carry alternate and additional text-and-attachments candidates so pre-existing checkpoint conflicts are not collapsed when recovery introduces another candidate. Candidate identity is the `(text, attachments)` pair, so equal text with distinct files/images remains separately selectable, while candidates identical in both fields are shown only once across authoritative, local, retained, and replacement state. The renderer independently preserves any newer local draft or staged attachments that had not yet reached main. If the replacement also carries an older host conflict, conflict presentation keeps three distinct candidates—the actual renderer draft, authoritative replacement text, and retained alternate—rather than selecting the wrong value. No old-identity UI operation can time out against the replacement host. Renderer editor patches, dialog responses, and panel input/resize/close carry the host instance and session epoch captured by their originating surface; main rejects them if either identity no longer matches, including when numeric panel IDs or editor revisions are reused. Panel render effects and input/acknowledgement sequence maps are keyed/reset by the full `(hostInstanceId, sessionEpoch, panelId)` identity, so even an old passive effect firing during a coalesced same-ID replacement cannot consume the new panel's first sequence. Planned worktree respawn applies the same UI retirement before detaching the old process and refuses to move while authoritative or retained work is active.
6. **Close:** `session.close` immediately fences normal ingress in main. If a live streaming host exists, main gives one best-effort ESC a bounded wait, then uses the private force-close protocol when possible. Cold, failed, missing, and unresponsive hosts take the same silent disposal path. Main always releases the in-memory session record; persisted session files and worktrees remain on disk. There is no renderer confirmation, close token, or cancellation path.


There is no fixed active-host cap and no generalized idle/LRU eviction. A renderer activation visit carries a unique token only when it causes a cold→live transition. Switching away within the two-second startup-cancellation window may release that exact token: main handles a release that races Pi discovery, waits for startup if necessary, takes a fresh direct snapshot, and retires the host through graceful disposal only if the visit had no user interaction and all submission, queue, editor, command, restoration, transition, dialog, panel, extension-published status/widget/working, and acknowledgement state is empty. The release-before-activation tombstone uses the same two-second expiry, so a delayed or failed discovery cannot retain cancellation state indefinitely. A pre-existing or older host is never assigned to the visit, returning cancels an in-flight release, and an unsafe release is abandoned rather than retried as a broad idle policy.

## Workspace search isolation

Workspace search is persisted-history-only and is not a second runtime authority. A disposable worker index may catalog JSONL for a currently registered workspace and known worktrees, including all saved branches and pre-compaction history, but JSONL supplies results and exact context. Querying and read-only preview do not touch `SessionRegistry`, activate a host, or change active workspace/session, branch, transcript, draft, attachments, editor revision, unread state, or runtime identity. Context is occurrence-exact and reports `ready`, `relocated`, `changed`, `removed`, `forbidden`, or `unavailable`; it never substitutes changed content.

Only explicit result opening enters normal stored-session opening and its activation-visit lifecycle. There remains no fixed active-host cap or generalized idle/LRU policy; search cannot reap a pre-existing host. See [workspace session search](session-search.md).

## History and transcript

Session discovery and complete history hydration read persisted JSONL asynchronously. Main converts histories cooperatively, yielding between bounded batches, and coalesces concurrent requests for the same canonical file path through cache publication (so settlement cannot open a duplicate-parse window). Initial hydration waits for an idle runtime and retries if the live transcript changes during the read, preventing persisted/live overlap from duplicating or overwriting racing events. Persisted transcript hydration walks the complete active on-disk branch across compaction markers: compaction limits the model's active context but does not remove earlier GUI scrollback. Main caches only a small LRU working set of converted complete histories so multiple large sessions do not remain resident indefinitely. A growing-file cache miss re-parses only during an idle re-hydration; sequential misses are bounded by those re-hydrations. Every `session.loadHistory` request carries the expected file, renderer history generation, and explicit host/epoch fields: a live runtime supplies both values, while an explicitly cold owner supplies a required `null`/`null` pair. Main admits the cold-owner form only for a non-activating record with no process or runtime identity. For live reads it requires ready/available, non-transitioning host/epoch identity, captures the actual registry record, process object, file, host, and epoch before file I/O, and validates the same lifecycle owner afterward; cold reads are therefore invalidated if activation begins before or during I/O, and live reads are invalidated as soon as transition starts. The renderer independently increments history ownership on file adoption, history seeding, and host/epoch replacement, and applies the complete history only while generation, file, and runtime identity still match. Initial open retries only after a store subscription observes a concrete live owner (or a genuinely new cold generation); it never retries an unchanged cold-owner request on an arbitrary short timer. Once ownership is stable, hydration waits for idle and rereads whenever the transcript changes during I/O. Each ownership/idle wait has a bounded timeout, and terminal failed/exited state ends the wait. The transcript reducer turns pi events and persisted entries into blocks, including custom entries and cache notices. Successful compaction moves the live block array by reference into immutable archive chunks and starts a small mutable tail at the marker; complete hydrated history is also retained as an archive chunk. The renderer owns that complete archive immediately, but mounts its presentation automatically in bounded newest-first React/DOM chunks, preserving the reading anchor as earlier chunks are prepended. This is cooperative presentation—not pagination or viewport virtualization—and prevents a single full-archive commit from freezing the renderer. Shared `FadeText` geometry is measured in one read-then-write frame batch so large transcript mounts cannot create per-label forced-layout cascades. Normal per-token copies and reconciliation remain confined to post-hydration or post-compaction live-tail work. It is intentionally separate from runtime authority: history loading, tree navigation, and transcript rendering cannot overwrite a current host snapshot.
