# Runtime services

Read this when changing browser preview, auth/login, embedded PTY, the pinned pi runtime, app update checks, subprocess environment, or native runtime dependencies.

- **Browser preview**: `npm run dev:renderer` loads `preview-stub.ts` which stubs `window.pivis` with a demo session and canned responses including streamed agent output.
- **Auth / subprocess env**: API keys stored in `~/.pi/agent/auth.json` using `proper-lockfile` for mutual exclusion with pi's token-refresh writes. Environment variables detected via `$SHELL -ilc env` (GUI apps don't inherit shell env). `getSubprocessEnv()` combines `process.env` + login-shell env for consistent subprocess PATH — used across git, pty, and node/pi subprocess spawns. Pi session spawns additionally merge `settings.piEnv` (advanced Settings UI, one `KEY=value` per line; applies on new spawn or `/reload`) after the login-shell env and before Pi-Vis-owned `PIVIS_*` controls. Atomic writes with tmp+rename, chmod 0600.
- **Login**: `/login` queries the active session's public `ModelRuntime` for provider and interactive method metadata, then dispatches an owner-bound `loginProvider` intent. The SDK host revalidates project trust plus the live provider/method immediately before `ModelRuntime.login()`. Its app-owned interaction surface supports OAuth URLs, device codes, progress/info, text/manual/select prompts, and password-style API-key input; prompt values remain short-lived UI responses and never enter Zustand, authority outcomes, replay baselines, or logs. Provider errors are sanitized. Successful runtime login writes through Pi's credential store and triggers the silent model-catalog refresh/readback path. Runtimes without the public login capability retain the embedded xterm.js terminal fallback.
- **Pinned pi runtime**: Pi-Vis bundles an exact pi version (`@earendil-works/pi-coding-agent` in `dependencies`) and never detects or updates a pi binary on the user's machine — upstream ships breaking SDK changes in patch releases, so the runtime only changes with a Pi-Vis release (audit `docs/compatibility/` first). `src/main/pi/pinned-pi.ts` resolves the bundled `dist/cli.js` + version; `settings.piBinaryPath` survives only as a test-only seam for e2e fake-pi fixtures. The app never exposes a pi self-update target. Two packaging constraints (both bit in practice): Electron's patched `fs` makes `existsSync` succeed for paths inside `app.asar`, so the resolver must remap any asar hit to the `app.asar.unpacked` mirror before a child consumes it; and electron-builder re-hoists pi's shrinkwrapped nested `node_modules` to top level at package time, so `electron-builder.yml` asarUnpacks **all** of `node_modules/**` (renderer deps are vite-bundled into `out/`; the packaged `node_modules` exists only for subprocesses/native modules) and the host's `resolvePiDependency` (bootstrap.mjs) searches ancestor `node_modules` dirs instead of assuming the nested layout. Packaged-app resolution cannot be covered by the repo-tree test suites — verify a release with a `--dir` build: spawn the unpacked cli, exercise the bootstrap imports, and activate a session in the packaged GUI.
- **Pi extension-package updates**: Separate from the pinned runtime and Pi-Vis app updates. A default-on, non-blocking launch check runs after 3s and respects `extensionUpdateCheckEnabled`; opening Settings also checks when no launch/previous result is cached, while the button always performs a fresh manual check. Main owns the cached status and a single-flight claim, so launch, Settings, and manual callers cannot duplicate one package-manager pass. Results publish through `extensionUpdates.status`, populate Settings with every installed user-scoped npm/git package plus its current/latest version, and add an update-only count to the Settings affordance. Checks use pi's public, best-effort `DefaultPackageManager.checkForAvailableUpdates()` in an environment-isolated worker, then enrich the installed catalog with package metadata and public npm/git commands; unavailable registry/remotes leave the latest version unknown instead of hiding the installed row. The worker creates `SettingsManager` with `projectTrusted: false`, so automatic checks never read or trust project-local package configuration. Mutations invoke the bundled CLI only as `pi update --extensions --no-approve` or `pi update --extension <source> --no-approve`; `--self` and `--all` are intentionally absent because the shipped pi runtime remains pinned. Main serializes mutations, and a timed-out CLI is terminated as a process tree so npm/git descendants cannot outlive the reported operation. IPC invokes are `extensionUpdates.status`, `extensionUpdates.check`, and `extensionUpdates.run`.
- **Pi-Vis app updates**: Separate from extension-package updates and from the pinned pi runtime. `src/main/app-updates.ts` uses Electron's built-in `autoUpdater` only for packaged macOS builds (`app.isPackaged`, `darwin`, not `PIVIS_DISABLE_APP_UPDATES=1`). The feed defaults to `https://update.electronjs.org/rsingapuri/pi-vis/${process.platform}-${process.arch}/${app.getVersion()}` and can be overridden with `PIVIS_APP_UPDATE_FEED_URL` for diagnostics. Background check runs at launch after 5s and respects `appUpdateCheckEnabled`. IPC channels are `appUpdate.status`, `appUpdate.check`, and `appUpdate.install`; events are emitted as `appUpdate.status`. When Squirrel.Mac reports `update-downloaded`, the renderer shows a restart-to-install prompt and calls `autoUpdater.quitAndInstall()` through `appUpdate.install`.
- **Renderer navigation**: BrowserWindow consumes Cmd/Ctrl+R and Cmd/Ctrl+Shift+R at `before-input-event`; Chromium refresh is not a supported lifecycle because it destroys generation-fenced renderer custody while SDK hosts continue running. Session `/reload` remains the explicit runtime reload path.
- **Dependencies**: `@homebridge/node-pty-prebuilt-multiarch` (native, externalized from main bundle, asarUnpack in electron-builder), `@xterm/xterm` + `@xterm/addon-fit` (renderer), `proper-lockfile` + `@types/proper-lockfile` (main).
