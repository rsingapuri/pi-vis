# Runtime services

Read this when changing browser preview, auth/login, embedded PTY, the pinned pi runtime, app update checks, subprocess environment, or native runtime dependencies.

- **Browser preview**: `npm run dev:renderer` loads `preview-stub.ts` which stubs `window.pivis` with a demo session and canned responses including streamed agent output.
- **Auth / subprocess env**: API keys stored in `~/.pi/agent/auth.json` using `proper-lockfile` for mutual exclusion with pi's token-refresh writes. Environment variables detected via `$SHELL -ilc env` (GUI apps don't inherit shell env). `getSubprocessEnv()` combines `process.env` + login-shell env for consistent subprocess PATH — used across git, pty, and node/pi subprocess spawns. Pi session spawns additionally merge `settings.piEnv` (advanced Settings UI, one `KEY=value` per line; applies on new spawn or `/reload`) after the login-shell env and before Pi-Vis-owned `PIVIS_*` controls. Atomic writes with tmp+rename, chmod 0600.
- **Login**: Native API-key sign-in (writes auth.json) + embedded xterm.js terminal for OAuth (spawns real `pi` in `node-pty`, watches `auth.json` for success detection).
- **Pinned pi runtime**: Pi-Vis bundles an exact pi version (`@earendil-works/pi-coding-agent` in `dependencies`) and never detects or updates a pi binary on the user's machine — upstream ships breaking SDK changes in patch releases, so the runtime only changes with a Pi-Vis release (audit `docs/compatibility/` first). `src/main/pi/pinned-pi.ts` resolves the bundled `dist/cli.js` + version; `settings.piBinaryPath` survives only as a test-only seam for e2e fake-pi fixtures. There is no pi/extension update checking or `pi update` runner in the app. Two packaging constraints (both bit in practice): Electron's patched `fs` makes `existsSync` succeed for paths inside `app.asar`, so the resolver must remap any asar hit to the `app.asar.unpacked` mirror before a child consumes it; and electron-builder re-hoists pi's shrinkwrapped nested `node_modules` to top level at package time, so `electron-builder.yml` asarUnpacks **all** of `node_modules/**` (renderer deps are vite-bundled into `out/`; the packaged `node_modules` exists only for subprocesses/native modules) and the host's `resolvePiDependency` (bootstrap.mjs) searches ancestor `node_modules` dirs instead of assuming the nested layout. Packaged-app resolution cannot be covered by the repo-tree test suites — verify a release with a `--dir` build: spawn the unpacked cli, exercise the bootstrap imports, and activate a session in the packaged GUI.
- **Pi-Vis app updates**: Separate from pi/extension updates. `src/main/app-updates.ts` uses Electron's built-in `autoUpdater` only for packaged macOS builds (`app.isPackaged`, `darwin`, not `PIVIS_DISABLE_APP_UPDATES=1`). The feed defaults to `https://update.electronjs.org/rsingapuri/pi-vis/${process.platform}-${process.arch}/${app.getVersion()}` and can be overridden with `PIVIS_APP_UPDATE_FEED_URL` for diagnostics. Background check runs at launch after 5s and respects `appUpdateCheckEnabled`. IPC channels are `appUpdate.status`, `appUpdate.check`, and `appUpdate.install`; events are emitted as `appUpdate.status`. When Squirrel.Mac reports `update-downloaded`, the renderer shows a restart-to-install prompt and calls `autoUpdater.quitAndInstall()` through `appUpdate.install`.
- **Dependencies**: `@homebridge/node-pty-prebuilt-multiarch` (native, externalized from main bundle, asarUnpack in electron-builder), `@xterm/xterm` + `@xterm/addon-fit` (renderer), `proper-lockfile` + `@types/proper-lockfile` (main).
