# Processes and IPC

## Process model

```
Renderer (React)  ⇄ typed Electron IPC ⇄  Main process  ⇄ child IPC ⇄  SDK-host AgentSession
```

Each active session has exactly one live authority: a `SessionHost` fork of `resources/pi-session-host/host.mjs`. The host imports the installed pi public SDK, creates the `AgentSession`, runs extensions and pi-tui surfaces, and owns command admission and state snapshots. There is no `pi --mode rpc` session path or fallback.

Main owns process lifecycle, session-file advisory locks, IPC validation, activation-visit retirement, and forwarding. The renderer owns presentation and sends typed requests through preload; it never reads Electron APIs directly.

## Authoritative runtime protocol

The host publishes direct `AgentSession` snapshots rather than synthetic liveness events. Each snapshot includes:

- `hostInstanceId`, `sessionEpoch`, and increasing `snapshotSequence`;
- direct `isStreaming`, `isIdle`, compaction, retry, bash, model, session, and queue values;
- host facts (submission, navigation, dialog, and custody state), extension catalog, and revisioned editor state.

Main rejects stale identity/epoch/sequence data. Snapshots have short availability leases (shorter while non-idle); expiry, an invalid envelope, or a real transport sequence discontinuity marks runtime state unavailable and requests a full resync. Ordinary control silence first starts one correlated full-snapshot probe and becomes unavailable only if its two-second deadline expires; polling jitter is not called a sequence gap, and identity retirement waits for eight seconds of continuous silence. Only valid control envelopes and successful correlated full-snapshot responses refresh this deadline—event, notification, panel, failed responses, and other non-control traffic cannot mask a wedged control channel. Each valid refresh advances a control generation, so a subsequently failing older silence probe cannot publish stale unavailability. A correlated gap-repair response validates host identity and snapshot epoch against its own envelope, then atomically adopts that epoch; this permits repair when the missing frame was the epoch transition itself. A renderer must gate controls and both native/unified submission context on `availability === "available"`, not infer readiness from status, stale booleans, or transcript lifecycle events. Main repeats this availability check immediately before dispatch so unavailable/transitioning runtimes cannot receive prompt ingress.

Host control records carry transport identity and sequence. Initial extension binding commits its ordered records and terminal snapshot in the `ready` envelope; session replacement uses the same atomic transition-batch model for the new epoch. Replacement begins only after a fresh idle/no-admission/no-custody/no-command check, and ordinary command ingress is rejected once transition starts. Transition-sensitive events, UI/panel records, and submission outcomes are buffered (blocking UI may be provisionally published to avoid deadlock), then committed with one terminal snapshot. This prevents partial startup or an old runtime record from overwriting new state.

## Submission, ESC, and UI acknowledgements

`session.submit` is the sole text/image submission API. The renderer includes an intent id, expected host id/epoch, editor revision, content, images, requested queue mode, and surface. The host reports an explicit disposition. During compaction or navigation it takes the request into FIFO custody; after a successful barrier, custody drains before later ingress. Cancelled navigation converts affected custody into review-required restoration rather than submitting it. Failed, stale, or uncertain admission stays recoverable rather than being optimistically represented as delivered text. A dispatched request is conservatively unknown until its correlated disposition arrives. If an uncancellable preflight remains unresolved, the host keeps it visibly pending and an admission watchdog retires the identity rather than allowing later idle work to overtake it.

Bare ESC uses `session.escape` with a request id and the renderer's captured host/epoch. Main returns `not_applicable` without dispatch for stale/unavailable runtimes, and returns `outcome_unknown` if transport is lost after possible dispatch; it never sends a delayed ESC into a successor. The host directly chooses the applicable operation (navigation, compaction, retry, streaming, bash, or editor) and returns its acknowledged disposition. Navigation, compaction, retry, streaming, and bash are cancellable; unresolved editor/submission preflight is explicitly `outcome_unknown` because Pi cannot safely cancel it. Interrupting streaming always emits a correlated queue-restoration record (empty markers are silently acknowledged by the renderer) and clears non-empty runtime queues under host authority. Because Pi exposes transformed text without attachment identities, the host retains every potentially queued original attachment while that queue mode remains non-empty and presents ambiguous originals unpaired rather than guessing by queue position. Main replays the record idempotently until the user dismisses the review card and the renderer acknowledges that restoration ID.

The renderer attaches with a monotonically increasing generation. Reattachment cancels blocking host UI from the old generation and requires an acknowledgement. Dialog responses and panel-close requests also carry operation ids and are accepted only after host acknowledgement. Panel input has per-panel sequences acknowledged by the host; main advances its cumulative acknowledgement only after host acceptance, and a gap is reported so the renderer can repair rather than silently advance. Editor changes and unsent attachment payloads use revisioned compare-and-apply patches, so renderer reload and explicit close do not lose component-local attachments. A unified-editor Enter carries its captured host editor revision and a main-assigned stable intent. Before any shared action runs, its renderer generation atomically claims the request with an unguessable claim ID and bounded deadline. Main replays only unclaimed requests while the same host identity/epoch still owns them. A claimed request that loses its renderer, host, or deadline becomes non-replayable review; main tombstones its stable intent, releases the host editor without restoring ambiguous text, and rejects late renderer acknowledgement. Renderer continuations also check the deadline/runtime after every await so late toasts, pickers, transcript writes, file hydration, and acknowledgements cannot land. Explicit review acknowledgement retires visible custody immediately; if a host submission promise is still unresolved, its protocol tombstone remains until that promise settles so acknowledgement cannot release a late result. Host failure converts unresolved unified text into review-required restoration and never executes it automatically against the replacement host.

## Lifecycle and capacity

Activation starts only an SDK host. Startup failure leaves the session failed and unavailable; it does not change execution mode. Unexpected host failure marks retained in-custody/consumed intents `outcome_unknown`, clears host panels, and may make one bounded SDK-host restart attempt. Reload runs as an intent-bearing in-host transition, is refused while the direct snapshot is non-idle, and settles against an explicitly named successor identity. Lost acknowledgement is review-only and is never replayed.

There is no fixed registry-wide active-process limit. The registry does not use a generalized idle/LRU policy because an apparently idle established host may still own extension background work. It performs one narrow startup cancellation instead: when a renderer visit itself caused a cold session to activate and requests release within two seconds, without any editor, submission, command, panel, dialog, ESC, restoration, or worktree interaction, `session.releaseActivationVisit` takes a fresh direct snapshot and reaps only that visit's host through its graceful disposal path. The release is refused for stale/pre-existing/older hosts, active/custodied/queued/editor/review/UI work, extension-published status/widget/working state, transitions, any open panel, or any interaction; returning during the check cancels it. Both normal and release-before-activation paths keep only a bounded two-second tombstone, so delayed discovery cannot turn the cancellation token into an unbounded lifecycle policy. Closing is two phase across both authority layers: `session.prepareClose` freezes main ingress and obtains a host-owned review checkpoint/token, while `session.confirmClose` validates both host and main mutation sequences before disposal. `session.cancelClose` reopens both layers. Force close remains explicit and marks unresolved ownership conservatively.

## Typed channels

Important invokes include `session.submit`, `session.escape`, `session.runtimeResync`, `session.rendererAttach`, `session.releaseActivationVisit`, `session.claimUnifiedSubmit`, `session.editorPatch`, `session.loadHistory`, `session.prepareClose`, and `session.confirmClose`. Main→renderer events include `session.runtimeState`, `session.submissionDisposition`, `session.queueRestoration`, and `session.uiAcknowledged`, alongside transcript, status, UI, and panel events. A delivered user `message_start` may carry its host-correlated GUI queue intent so ownership never depends on extension-transformed text. See `src/shared/ipc-contract.ts` and `src/shared/pi-protocol/runtime-state.ts` for the schemas.

Extension custom and unified pi-tui panels are host-rendered ANSI surfaces bridged through `session.panelEvent`, `session.panelInput`, and `session.panelResize`. The host is color-agnostic: it emits stable indexed ANSI roles and the renderer resolves them against the active scheme.
