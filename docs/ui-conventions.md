# UI conventions and key patterns

## Key Patterns

- **ESC-to-interrupt (acknowledged host operation)**: A single capture-phase `useGlobalEscapeInterrupt` handler in `App.tsx` handles bare ESC only after overlay claims defer it. It calls `session.escape`; the SDK host directly selects navigation, compaction, retry, streaming, bash, or editor handling and returns an acknowledged disposition. The renderer never infers abortability from transcript events or synthetic liveness. Modified/IME Escape and non-active sessions are ignored. `useEscapeClaim(open)` remains ref-counted and uses `useLayoutEffect`: autocomplete, dialogs, pickers, panels, viewers, menus, and viewport-mode unified overlays claim the first Escape. Thus autocomplete closes before a later bare Escape can request an interrupt, and a pi-tui overlay closes before an interrupt request. Renderer-generation and UI-operation acknowledgements ensure a late ESC/UI action from a replaced renderer cannot affect the current host epoch.

- **Shift+Enter = newline in the unified editor** (Kitty keyboard protocol): the unified-TUI editor (and `custom()` panel inputs) distinguish Shift+Enter from Enter via the Kitty protocol — Shift+Enter inserts a newline (multiline editing), Enter submits. This requires BOTH halves to be negotiated (see the “Keyboard protocol negotiation” note in `docs/architecture/processes-and-ipc.md`): the renderer enables `vtExtensions.kittyKeyboard` on xterm 6.1 (`basePanelTerminalOptions()`), and the host performs the matching push/query/filter handshake over the panel wire so pi-tui decodes the CSI-u (`\x1b[13;2u`). Bracketed paste is enabled in the same handshake, so a multiline PASTE inserts lines and never submits on a newline. If kitty can't be negotiated (old pi-tui, or xterm fails to grant it), the editor degrades to today's behavior (Shift+Enter indistinguishable from Enter) rather than a half-enabled state.

- **Diff edit card ESC ownership**: while an inline edit session is open in the diff viewer, the card's textarea Escape uses `stopPropagation` (the CommentEditor pattern) so neither the viewer-close branch nor the global interrupt sees it; a dirty Escape raises the `ConfirmDialog` ("Discard edit?"). The viewer's own Escape branch and backdrop click, while a session exists, do NOT close the viewer — they bump `editCancelNonce`, which the card watches to run its cancel flow (confirm if dirty). See `docs/architecture/diff-editing.md`.

- **Branded types** for IDs: `SessionId`, `RpcRequestId` are `string & { __brand: "..." }` — prevents accidental mixing
- **Zod schemas everywhere**: All protocol types, settings, session files validated with Zod. Schemas live in `shared/` and are the single source of truth.
- **Pure transcript reducer**: `transcript.ts` is a pure function — no side effects, no store access, no in-place mutation. Easy to test. The per-token streaming path (`patchBlock`) is still pure: it returns a fresh array, but copies only the spine and replaces the one streamed slot, so it dodges the O(n) per-element `.map` that made streaming O(n²) on long sessions without sacrificing immutability or referential integrity.
- **Transcript bottom pinning**: `TranscriptView` follows the bottom unless the user actively scrolls up. Layout-driven scroll/viewport changes (notably xterm-sized custom/unified TUI panels replacing the Composer) must not clear the pinned state. User intent is tracked from wheel/touch/scroll-key input; programmatic/layout scroll events are corrected back to bottom while pinned. `panel-sizer.ts` emits a bubbling `pivis:composer-slot-resize` event after it applies JS heights so the transcript can re-pin immediately instead of waiting for the next streamed token. Covered by `tests/render/transcript-bottom-pin.spec.mts`.
- **Unified TUI focus**: mounting a factory widget may autofocus its terminal only when no other typing control owns browser focus. In particular, a panel appearing in the background must not blur an in-progress session rename; clicking the terminal still focuses it explicitly.
- **Unified TUI height negotiation**: intrinsic factory-widget output may grow beyond the visible card cap; `panel-sizer.ts` measures the complete xterm buffer (including a first oversized frame's scrollback), grows ordinary content with a blank sentinel row, and switches extremely tall frames to xterm's internally virtualized scrollback so all retained rows remain reachable without thousands of DOM viewport rows. A widget whose output grows again to exactly fill that larger probe is viewport-coupled even when it cannot send `panel_mode: viewport`; pin it to the visible cap instead of feeding an upward-expanding resize loop, but release that pin if later output enters scrollback. Explicit pi-tui overlays still use host-signalled viewport mode. A presentation threshold must switch scrolling owners rather than clip: an exact-boundary frame cannot receive a grid sentinel, while a larger frame must remain reachable through xterm's internal viewport. Detect implicit viewport coupling from two consecutive matching grid/content growth deltas (including `rows + k` widgets), not one filled repaint; the resize-storm breaker remains defense-in-depth. Covered by `tests/render/unified-panel.spec.mts`.
- **Transcript style setting**: Settings → Interface offers `Verbose` (the default, exact chronological block rendering) and `Compact`. Compact is display-only: it derives grouped render items from the existing transcript without changing reducer state/history, so toggling immediately re-renders old sessions both ways. Compact leaves user blocks, assistant text prose, and final errors visible, while folding thinking/tool/activity blocks that occur between output prose into an inline disclosure summary (e.g. `Thinking, 10 tool calls`). Retryable provider errors stay inside the surrounding folded activity instead of splitting out a new compact group. Expanded collapsed content is indented under a subtle left hairline; active groups show the same spinner language as streaming tool/thinking rows, but that spinner clears as soon as visible output after the folded activity begins.
- **Transcript activity cards**: operational/context events (`compaction`, `custom_message` notices, `/session` output) render with the same `.tool-card` shell as normal tool calls, not as tiny freestanding text or a separate visual system. They keep chronological placement in the transcript, use a `context`/`notice` tool-name label + FadeText subject, show a short markdown preview, and expand in-place for the full details.
- **Tool output inspection**: expanded tool/bash cards should show all retained output Pi-Vis has, not an arbitrary expanded-preview slice. Keep the transcript compact when collapsed, but use GUI affordances in expanded state: result metadata/truncation details in small disclosure cards, full command/subject text via `FadeText`, and a bounded virtualized output well with top/bottom fades plus a “Copy all” action. Preserve pi result `details` (for example `truncation`, `fullOutputPath`, and `diff`) in transcript state/history so the UI can explain when pi itself already truncated upstream.
- **Fire-and-forget UI requests**: `notify`, `setStatus`, `setWidget`, `setTitle`, `set_editor_text` are handled as side effects in `addUiRequest` without awaiting a response. Dialog types (`select`/`confirm`/`input`/`editor`) block pi until the renderer responds.
- **Notifications** (`components/notifications/NotificationStack.tsx`): session toasts are now retained in per-session notification history but presented as transient in-session cards. `NotificationStack` mounts inside `.transcript-region` (the positioned wrapper around `TranscriptView`), not in the whole `.app__session`, so cards overlay transcript content without reserving horizontal space and cannot cover the composer/dock/status area. A new notification shows a top-right stacked-card preview for ~6.5s (paused while hovered), then fades out unless the user clicks it; clicking anywhere on the preview bubble or the title-bar `NotificationBellButton` (rendered only while notifications exist) sets `notificationPanelOpen` and expands the full scrollable list with top/bottom fade overlays so text can be selected/copied. Notification type is indicated by a compact semantic marker (no headers or colored borders). The expanded history claims ESC, closes on bare Escape, and closes on outside click (including transcript clicks) so it cannot accidentally abort a streaming agent. Dismiss removes one toast (`dismissToast`); Clear all uses `clearToasts` and closes the panel. Cleared notifications do not reappear, but future notifications create a fresh transient preview.
- **Diff viewer UI**: capped changed-file counts are labeled as lower bounds (for example `>499`) anywhere the viewer/header badge would otherwise show the cap as an exact number. The cap applies to normal browsing sections, never search discovery: find progressively searches the complete descriptor manifest, labels partial counts as `N+` with file progress, reports unavailable binary/too-large/failed files, and navigates distant matches through a small targeted row island rather than mounting their entire prefix. File headers are sticky within the diff scroll area and use a subtle lower fade so long files keep their filename context while rows scroll underneath. The selected base branch is remembered per live session for the current app run, but is cleared back to HEAD if the branch list no longer contains that ref.
- **Composer file attachments** (`components/composer/Composer.tsx`): the `+` button is "attach files", not "attach images", and stays available for text-only models. Non-image files render as compact file tiles in the composer (rather than dumping paths into the editable text); on prompt submit their paths are prepended to the sent message (`/path/to/file\nuser text`). Image-capable models keep Pi-Vis's enhanced image previews/base64 RPC payloads; when the current model does not support images, picked image files fall back to generic file tiles and one warning toast per picker selection explains the downgrade. Diff code comments share this compact-tile language: one comment tile appears when the active session has pending diff comments, shows the count, and clears only after a successful ordinary prompt send prepends those comments to the outbound prompt. Slash commands—including extension/template/skill commands sent through the prompt transport—never receive or clear staged comments/files/images; only their command text is consumed. In the diff viewer, saved comments render as inline review threads under their anchored new-side line in both unified and split views (not just as gutter glyphs), with the gutter bubble centered on the row. Comments are renderer-local but durably stored in `localStorage`, keyed by live session id with a session-file handoff when a resumable tab is closed/reopened; each comment stores a stable id plus saved new-side line text so diff refresh can relocate anchors when that line moves uniquely or mark them stale when it cannot.
- **Map immutability**: Zustand stores create new `Map` instances on every update (never mutate in-place) since Zustand uses reference equality for selectors.
- **CSS**: Custom CSS with BEM naming (`composer__input-row--bash`). No CSS framework. CSS modules co-located with components. `global.css` defines an app-wide focus policy: pointer-driven focus has no outline (`:focus:not(:focus-visible)`), keyboard focus shows a lavender `:focus-visible` ring. **Exception**: the slot that the Composer occupies (`.ext-dialog-slot`, `.custom-panel`, `.picker-slot`) suppresses `:focus-visible` on internal elements — the Composer's mauve input ring is the *only* focus affordance in that slot, and components that replace the Composer (extension dialogs, custom panels, built-in pickers) inherit no focus rings. The dialog's option list uses a JavaScript-managed `.ext-dialog__option--highlighted` state for arrow-key navigation, and text fields use the caret — both are intent-revealing, not focus-revealing.
- **Truncation = FadeText, not ellipsis** (`components/common/FadeText.tsx`): every single-line label that can outgrow its box (sidebar workspace/session names, session-header title + model label, status-bar model, tool-card subjects, composer suggestion descriptions, dock/update-banner names, widget lines, diff-tree dir/file labels, diff file-header dirnames, picker item names, trust cwd, tree-viewer rows, branch-dropdown labels/trigger) renders inside `<FadeText>` instead of `text-overflow: ellipsis`. Overflow is JS-measured (ResizeObserver on both the clipping outer span and the content inner span; jsdom-guarded for unit tests); when overflowing, the trailing edge fades via a mask and hovering the text — or an ancestor marked `.fade-scope` (rows/buttons pass their whole hover area) — glides the text sideways to reveal the tail at a capped readable speed (duration proportional to hidden width with no max-duration acceleration for extremely long labels), gliding back on leave. `head` prop flips the direction for values whose TAIL matters (directory paths): tail visible at rest, leading edge fades, hover reveals the head — this replaced the old `direction: rtl` left-ellipsis hack (and its slash-reordering workaround) in the diff viewer. **One FadeText per truncatable value**: the diff file header wraps the ENTIRE path (dim dirname span + bright basename span as children of a single head-mode FadeText) — splitting one string across two adjacent FadeTexts produces a double fade with two tiny disconnected glides (and, for paths, a stray leading `/` once the dirname collapses). `pre` preserves whitespace (dock widget lines). Site CSS keeps layout/color (`flex`/`min-width: 0`/color) and must NOT re-add `overflow/white-space/text-overflow`; FadeText owns clipping. Respects `prefers-reduced-motion` (no glide).
- **Scrollbars**: one app-wide treatment in `theme.css` — a slim rounded pill floating in a transparent gutter (fixed px width; transparent 3px border + `background-clip: padding-box` inset the thumb), `surface-2` → `surface-3` on hover. Components must NOT restyle scrollbars (`scrollbar-width`/`scrollbar-color` also override the webkit styling in Chromium 121+, so don't use them either); the sanctioned exceptions hide the bar entirely because an edge-fade mask is the scroll affordance (`.transcript-view`, `.sidebar__workspaces`, `.notification-stack__list`).
- **Icons = shared SVG set, never text glyphs** (`components/common/icons.tsx` + `.icon`/`.icon-btn` in `theme.css`): every chrome glyph (`▾ ✓ × ▼ ▲ ⑂` and the old border-drawn chevrons) is a stroke SVG from the shared set — `IconChevronDown/Up/Right`, `IconCheck`, `IconClose`, `IconBell`, `IconBranch` — 12×12 viewBox, 1.5px rounded strokes, `currentColor`, 1em default sizing (`.icon` also handles baseline alignment for inline use). Interactive icons sit in the global `.icon-btn` ghost chrome (1.714rem square hit target, no resting fill, `surface-2` lift on hover, built-in `:disabled`); components add their BEM class only for placement/size overrides (e.g. the Dock's compact 1.286rem dismiss). Viewer-specific glyphs (search, refresh, case, label) stay local to their host but follow the same geometry. Dropdown-trigger carets are 0.714em at 0.8 opacity (`.session-header__caret`, `.branch-dropdown__caret`).
- **Popup card language** (dropdowns/pickers/suggestions): every floating list — model + thinking dropdowns, BranchDropdown, slash suggestions, `/model`-style pickers, the ContextMeter card — is a raised `--surface` card (`--radius-lg`, hairline border, `--elevation-2`) whose **inner list scrolls with padding** (`0.143rem var(--space-2) var(--space-2)`, flex column, 0.143rem gap) and whose rows are **rounded inset rows** (`--radius-md`, transparent 1px border; highlighted = `--surface-2` fill + `--surface-3` border) — never full-bleed strips butting into the card's rounded corners. Search inputs are fixed (non-scrolling) headers; pinned footers (BranchDropdown's remote checkbox) are full-bleed bars under a hairline `border-top`. When Settings → Interface → `Group models by provider` is enabled, the header model dropdown uses inline expandable provider groups (not side flyouts, because the picker is constrained near the title-bar edge); typing in the search field intentionally falls back to the normal flat filtered model list. Slash-suggestion source badges are a colored dot + ghost text (dot hue = source: accent/built-in, success/extension, info/prompt, warning-soft/skill), not outlined pills. The image lightbox is a viewport-bounded modal card (`dvh` sizing with mobile-safe padding); the header stays fixed and the image pane owns overflow scrolling with `object-fit: contain`/logical max sizes so unusual image aspect ratios remain fully reachable instead of being clipped by the modal shell.
- **Stats live in the ContextMeter** (`session-header/ContextMeter.tsx`): the title-bar context ring opens a dropdown card with the context headline + linear meter (accent → `warning-soft` ≥80% → `danger` ≥90%), then token rows in order: input, output, and only non-zero/defined cache read, cache write, and cache hit rate. The StatusBar under the composer is deliberately minimal — one `path · model` line (both FadeTexts, no wrapping) plus extension ANSI segments; it must NOT re-grow the old raw sigil line (`↑3.2K ↓290 R2.2K …`) that duplicated the ContextMeter. The Composer textarea intentionally has no visible placeholder or discovery-hint text; it keeps the accessible `aria-label="Message pi"` for screen readers.
- **Design tokens** (`theme/theme.css` `:root`): a flat, modern token system that every component composes from — `--space-1…10` (rem spacing scale), `--radius-sm/md/lg/xl/pill` (soft, consistent corners), `--leading-*`/`--tracking-*` (type), and crucially the separation tokens: `--border` / `--border-faint` (both `--surface`) and `--border-strong` (`--surface-2`) for hairline edges, `--surface-raised` (`--surface`) / `--surface-inset` (`--bg-sunken`) for in-flow depth, and `--elevation-1/2/3` shadows for floating layers (menus → `-2`, modals → `-3`). The interface font family is app-owned (not user-selectable) so controls can be tuned against stable font metrics; Settings may scale the interface size, while code/terminal font family and size remain user preferences. `--font-accent` (Fraunces) is intentionally reserved for the workspace label, the active session title in the title bar, and app-owned modal headers only; these usages run slightly larger than adjacent UI text and use `--font-accent-tracking` for subtle tightening. Do not apply it to sidebar session rows, picker titles, or body/control text. Transcript-specific type uses `--font-thinking` (IBM Plex Serif italic) for thinking blocks and `--font-reading-heading` (IBM Plex Serif non-italic) for assistant Markdown headings outside thinking; both use `--font-size-serif` (`calc(var(--font-size-base) * 1.07)`) so IBM Plex Serif renders at 1.07× the Inter body text size from the global base, independent of parent font-size inheritance; monospace code embedded inside thinking blocks uses the root-derived `--font-size-code-root` so it does not inherit the serif bump. Transcript Markdown is compositional: containers set voice/tone (`.transcript-block__content`, blockquotes, `.thinking-block`) and structural elements such as headings inherit that tone while providing a compact but visibly stepped h1–h6 hierarchy. Built-in pickers (`/model`, `/fork`, `/resume`) are **not** modals — they replace the Composer in the flex slot (`.picker-slot`, an in-flow raised `--surface` card at `--elevation-1`), matching extension dialogs and custom panels. The look leans on hairlines + surface elevation + spacing instead of high-contrast outlines and box-in-box nesting — e.g. tool cards are a raised `--surface` card over a recessed `--bg-sunken` well, no inner border. There is **no MCM/mid-century vocabulary** anymore (the previous design language was removed).

## Queued transcript bubbles

Queued steering/follow-up messages render between delivered transcript blocks and the working row. They are right-aligned, user-like pending bubbles with a faint dashed semantic-token border, reduced opacity, and captions (`Steering — queued` / `Follow-up — queued`). They are not transcript history. Visible ownership is exclusive: while an optimistic normal user bubble owns a pending authoritative echo, the matching authoritative queue item is suppressed by its host-correlated submission intent; when `message_start` delivers queued input, that same intent transfers ownership atomically to the normal transcript bubble. Text is never an ownership key because extensions may transform both queued and delivered prompt text. Queue snapshots and user echoes must therefore never expose both projections of one submission in the same renderer state.

If a host interruption makes queued-input custody ambiguous, a compact **Review interrupted message** card explains that retrying may duplicate already-processed input and offers explicit restore/dismiss actions. It uses the normal raised surface and only a small semantic warning marker—never a warning-color card fill. Attachment caveats appear only when recoverable attachments exist, and empty custody markers are acknowledged without rendering an actionless warning card. The same surface renders **Review interrupted command** when an effectful command crossed host IPC but lost its acknowledgement; it describes the operation, never auto-replays it, and does not offer “Restore to Composer” unless the restoration is ordinary queued text.
